# Подробное Техническое ТЗ

## 1. Scope
Документ определяет требования к разработке мобильной F2P игры block puzzle на Flutter + Flame:
- функциональные требования;
- нефункциональные требования;
- интеграции;
- критерии приемки;
- ограничения на MVP и планы расширения.

## 2. Product Scope по этапам
1. Prototype (Этап 0):
- базовое поле 8x8;
- drag&drop фигур;
- проверка валидности;
- очистка линий;
- подсчет очков.

2. MVP (Этап 1):
- полноценный endless режим;
- onboarding/tutorial;
- базовые эффекты и звук;
- реклама (banner/interstitial/rewarded revive);
- аналитика ключевых событий.

3. Soft Launch (Этап 2):
- дневные цели, streak, базовые ачивки;
- A/B для генератора фигур и ad pressure;
- IAP (ad-free + cosmetic starter packs);
- live balancing через remote config.

4. Scale (Этап 3+):
- дополнительные режимы;
- продвинутый метагейм;
- расширенная liveops система.

## 3. Functional Requirements

### FR-CORE (геймплей)
- FR-CORE-001: Игра использует квадратное поле 8x8 в MVP.
- FR-CORE-002: Игрок получает набор из 3 фигур и обязан разместить все до генерации следующего набора.
- FR-CORE-003: Фигура ставится только при полном легальном размещении без пересечений.
- FR-CORE-004: Полная строка или столбец очищается автоматически после хода.
- FR-CORE-005: Начисление очков учитывает количество очищенных линий и комбо-множитель.
- FR-CORE-006: Game Over наступает, если ни одна из текущих фигур не может быть размещена.
- FR-CORE-007: Игрок может немедленно начать новый раунд после Game Over (1 tap restart).

### FR-UX (интерфейс и опыт)
- FR-UX-001: Экран старта партии не должен содержать лишних шагов.
- FR-UX-002: Drag preview показывает валидность установки фигуры до отпускания.
- FR-UX-003: Невалидная установка визуально и тактильно объясняется игроку.
- FR-UX-004: После очистки линии есть аудио-визуальный burst feedback.
- FR-UX-005: HUD всегда показывает текущий счет, рекорд, текущий combo-state.

### FR-PROG (прогрессия и мета)
- FR-PROG-001: Локальное хранение лучшего счета.
- FR-PROG-002: Дневные задания (MVP+1).
- FR-PROG-003: Streak механика ежедневного возврата (Soft Launch).
- FR-PROG-004: Коллекция тем/скинов (Soft Launch+).

### FR-MON (монетизация)
- FR-MON-001: Banner доступен в игровых/меню экранах согласно UX-ограничениям.
- FR-MON-002: Interstitial показывается по frequency cap и контекстным триггерам.
- FR-MON-003: Rewarded revive доступен после Game Over.
- FR-MON-004: Rewarded hint/undo доступен в ограниченных сценариях сложности.
- FR-MON-005: IAP ad-free отключает баннер и interstitial, но не forced value rewarded.

### FR-DATA (аналитика и эксперименты)
- FR-DATA-001: Логируются ключевые события с обязательными параметрами.
- FR-DATA-002: Offline queue для событий, последующая отправка при сети.
- FR-DATA-003: Remote Config определяет параметры сложности/рекламы/онбординга.
- FR-DATA-004: Игрок закрепляется за A/B вариантом детерминированно.

### FR-OPS (операционное управление)
- FR-OPS-001: Feature flags позволяют включать/выключать функционал без релиза.
- FR-OPS-002: Конфигурации версионируются.
- FR-OPS-003: Все изменения сложности и ad pressure журналируются.

## 4. Non-Functional Requirements

### NFR-PERF
- NFR-PERF-001: Цель 60 FPS (не ниже 55 FPS на low-mid Android).
- NFR-PERF-002: Input latency не должна ощущаться пользователем (>90 ms недопустимо).
- NFR-PERF-003: Время холодного старта до интерактива <= 4 сек на целевых девайсах.

### NFR-RELIABILITY
- NFR-REL-001: Crash-free sessions >= 99.5%.
- NFR-REL-002: Оффлайн режим базового геймплея обязателен.
- NFR-REL-003: Некорректные события не должны ломать игру.

### NFR-SECURITY
- NFR-SEC-001: Токены и ключи SDK не хранятся в открытом виде в репозитории.
- NFR-SEC-002: Минимизировать PII; идентификаторы пользователей анонимизированы.
- NFR-SEC-003: Конфигурации monetization и A/B подписываются/валидируются.

### NFR-MAINTAINABILITY
- NFR-MAIN-001: Модульная архитектура с четкими boundary.
- NFR-MAIN-002: Unit тесты для core domain logic.
- NFR-MAIN-003: Контрактные тесты для data/event schema.

## 5. Архитектурные решения
- Клиент: Flutter + Flame.
- Слой доменной логики не зависит от SDK монетизации и UI.
- Монетизация, аналитика и remote config подключаются через адаптеры/интерфейсы.
- Динамический баланс сложности управляется server-driven конфигом.

## 6. Acceptance Criteria (MVP)
- Игрок может сыграть минимум 10 раундов подряд без критических ошибок.
- Метрики `game_start`, `game_end`, `move_made`, `ad_impression`, `ad_rewarded` приходят консистентно.
- Реклама не нарушает базовый цикл (нет soft-lock, нет падений FPS ниже минимального порога).
- Есть возможность отключить ad placements флагом server-side.

## 7. Out of Scope для MVP
- PvP, real-time мультиплеер.
- Сложный narrative режим.
- Генеративный UGC редактор уровней.
- Полноценный live-events конструктор.

## 8. Артефакты разработки
- Архитектурная схема слоев и модулей.
- Event taxonomy + data dictionary.
- Remote config schema + default fallback values.
- План экспериментов первого soft-launch цикла.
- Набор QA smoke/regression сценариев.
